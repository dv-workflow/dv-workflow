Claude là gì?

Claude là AI assistant của Anthropic — được thiết kế để helpful, harmless, and honest.

  • Được train bởi Anthropic (2021, safety-focused AI lab)
  • Large Language Model (LLM) — hiểu và sinh text tự nhiên
  • Có thể viết code, phân tích, debug, review, giải thích
  • Context window lên đến 200K tokens (~150K words)
  • API-first — tích hợp được vào workflow
Anthropic được thành lập bởi ex-OpenAI researchers. Focus vào AI safety. Constitutional AI — Claude được train để tự phê bình và từ chối harmful requests. Context window 200K = có thể đọc toàn bộ codebase vừa trong 1 lần.

Claude Model Family

Model Speed Intelligence Cost Best for
Haiku ⚡⚡⚡ Fastest ★★★ $ Simple tasks, high volume, realtime
Sonnet ⚡⚡ Fast ★★★★★ $$ Daily dev work — default choice
Opus ⚡ Moderate ★★★★★+ $$$ Complex reasoning, architecture

💡 Rule of thumb: Dùng Sonnet cho daily work. Upgrade Opus khi task cần deep reasoning.

Hiện tại latest: claude-sonnet-4-6 và claude-opus-4-6. Haiku tốt cho: autocomplete, quick Q&A, high-volume batch processing. Sonnet: code generation, debugging, review — best value. Opus: system design, complex refactoring, ambiguous problems.

Agentic Mode — Sức mạnh thực sự

Claude không chỉ trả lời — nó có thể tự hành động trong môi trường thực

Cách hoạt động

📋 Nhận task từ user
↓
🔍 Research: đọc files, grep code
↓
🧠 Plan: phân tích, lập kế hoạch
↓
⚙️ Execute: edit files, run commands
↓
✅ Verify: chạy tests, check output
↓
📊 Report kết quả cho user

Tools có sẵn

📂Read/Write/Edit — Đọc, ghi, sửa files
🔎Grep/Glob — Tìm kiếm trong codebase
💻Bash — Chạy commands, tests, builds
🌐WebSearch/WebFetch — Tìm kiếm web, đọc docs
🤖Agent — Spawn sub-agents song song
🔌MCP Tools — Extensible với custom tools

💡 Agentic = Claude có thể hoàn thành task multi-step mà không cần prompt từng bước

Đây là điểm khác biệt lớn nhất giữa Claude API thuần và Claude Code agentic mode. Ví dụ thực tế: "Add unit tests for UserService" → Claude tự đọc UserService, tìm methods, viết tests, chạy, fix nếu fail, commit. Sub-agents: một task lớn có thể spawn nhiều agents chạy song song — research agent, implementation agent, review agent. MCP (Model Context Protocol): giao thức mở để thêm custom tools — database queries, Jira integration, Slack, etc.

Extensibility

⚡Slash commands — Custom workflows: /my-deploy, /pr-review
📋CLAUDE.md + Rules — Code style, conventions persist across sessions
🔌MCP Servers — Connect DB, Jira, Slack, GitHub trực tiếp

Điểm mạnh so với Copilot/Cursor

  • Đọc toàn bộ codebase → hiểu đúng context
  • Edit files, chạy tests, git — end-to-end
  • Permission model rõ ràng — user approve từng action
  • CLAUDE.md = "onboard" Claude vào project của bạn
  • Sub-agents chạy song song cho tasks phức tạp
  • Open extensibility qua MCP protocol
Claude Code khác Copilot: không chỉ autocomplete mà thực sự hiểu project context và thực hiện multi-step tasks. CLAUDE.md: viết 1 lần, Claude đọc mỗi session — giống như onboarding doc cho AI. MCP (Model Context Protocol): open standard, cộng đồng đang build rất nhiều MCP servers. Permission model: Claude hỏi trước khi write/delete files — an toàn hơn auto-apply.

Prompt Engineering Basics

1. Clear Context

❌ "Fix this bug"

✅ "In src/auth/login.ts, the JWT
   validation fails when token has
   no expiry. Fix using our existing
   verifyToken() helper"

2. Specify Output

✅ "Return only the fixed function,
   no explanation needed"

✅ "Give me a markdown table comparing
   these 3 approaches"

3. Examples Help

✅ "Write tests like this example:
   [paste existing test]
   Now write tests for UserService"

4. Role Framing

✅ "You are a security engineer.
   Review this auth middleware
   for vulnerabilities"
Prompt engineering không phải magic — chỉ là communication skill. Tốt nhất: treat Claude như một senior dev mới join team. Cho context, give examples, specify what you want. Iteration is fine — nếu không đúng, refine và ask again.

Use Cases & Best Practices

High-value tasks

★★★
Code Review — Tìm bugs, security issues, performance problems
★★★
Test Generation — Generate unit/integration tests từ existing code
★★★
Debugging — Trace errors, explain stack traces, suggest fixes
★★
Refactoring + Docs + Boilerplate — Extract functions, generate README, CRUD

Best practices

01
Small focused prompts — 1 task = 1 prompt, đừng mix
02
Review mọi output — Treat như code của junior dev
03
Iterate với feedback cụ thể — "Fix X, keep Y, add Z"
04
Paste context đầy đủ — Error message, relevant code, expected behavior
05
Dùng Agentic mode — Cho tasks multi-step, Claude tự research + execute
High value: nơi Claude tiết kiệm nhiều thời gian nhất — code review không mệt, test gen viết 80% đúng. Best practice quan trọng nhất: review output. Claude không biết business logic riêng của project bạn. Agentic mode đặc biệt tốt cho: "add tests to all services in this module" — 1 prompt, Claude tự làm hết.

Điểm yếu cần lưu ý — Claude

⚠️

Hallucination — Nguy hiểm nhất

❌ Thực tế Claude tạo function getUserById() gọi đến db.findUser() — method này không tồn tại trong codebase của bạn
✅ Phòng tránh Review mọi generated code như review junior dev. Run tests. Grep để verify function names thực sự tồn tại.
🌀

Context Drift — Trong sessions dài

❌ Thực tế Sau 50+ turns, Claude "quên" conventions ban đầu, bắt đầu mix coding styles hoặc đi lạc khỏi requirement
✅ Phòng tránh Break tasks thành sessions nhỏ. Tóm tắt context quan trọng khi bắt đầu session mới. Nhắc lại key constraints khi cần.
📅

Knowledge Cutoff + Library Versions

❌ Thực tế Claude suggest API của React 18 khi project đang dùng React 19, hoặc dùng deprecated pattern của NestJS v9
✅ Phòng tránh Ghi rõ version trong CLAUDE.md (Claude Code) hoặc paste relevant docs vào context. Luôn verify với official docs.
🔐

Data Privacy — Không paste sensitive data

❌ Tránh Paste API keys, database credentials, customer PII, hoặc proprietary business logic vào prompt
✅ Rule Dùng placeholder (SECRET_KEY, USER_DATA). Enterprise tier có data isolation. Đọc data governance policy.
Hallucination: case thực tế — Claude generate unit test gọi method không tồn tại, test pass khi mock nhưng fail ở integration. Luôn run actual tests. Context drift: nếu thấy output quality giảm → start new session với context summary. Privacy: nhiều công ty có policy về AI tools. Đảm bảo team biết và tuân thủ. Knowledge cutoff hiện tại: August 2025. Mọi thứ sau đó Claude không biết.

Demo — Claude Code trong thực tế

Scenario: Fix bug + viết test cho một API endpoint

1

Mô tả bug

claude "The POST /api/users endpoint returns 500
when email already exists. It should return 409
Conflict. Fix it and add a test."
2

Claude tự research

▶ Reading src/api/users.controller.ts ...
▶ Reading src/services/user.service.ts ...
▶ Found: no duplicate email check before insert
▶ Reading users.controller.spec.ts ...
3

Fix + test

// Thêm check trước khi insert
const exists = await this.userService.findByEmail(dto.email);
if (exists) throw new ConflictException('Email already exists');
// + viết test case cho 409 scenario
4

Verify

▶ Running: npm test -- users.controller.spec
✓ POST /users → 201 Created
✓ POST /users duplicate email → 409 Conflict
All tests passed.
Demo live nếu có môi trường sẵn, hoặc walk through scenario này. Key insight: Claude không chỉ "suggest" — nó đọc actual code, fix đúng pattern của project, viết test theo existing style. Toàn bộ flow từ bước 1 đến khi test pass: ~2 phút. Manual: 15-30 phút. Nhấn mạnh: dev vẫn phải review code trước khi commit.

DW Toolkit

Workflow cho Dev Team

Làm việc với AI có hệ thống, reproducible, và scalable

DW = Dev Workflow. Toolkit này sinh ra để giải quyết vấn đề: "Dùng Claude thế nào cho đúng, consistent, không phụ thuộc vào từng dev biết prompt hay không"

Vấn đề cần giải quyết

😵
Mỗi người dùng Claude một kiểu — Không có standard workflow, output inconsistent
🤷
Không có handoff — Session mới mất context, AI không biết đang ở đâu
🏃
AI code thẳng không có plan — Vào execute ngay, scope creep, bugs khó track
📊
Không track được effort — AI làm bao nhiêu? Estimate có đúng không?
🔒
Thiếu gates — Không có checkpoint để human review trước khi AI execute
Những vấn đề này phổ biến khi team bắt đầu dùng AI coding tools. DW Toolkit giải quyết bằng cách chuẩn hóa workflow với AI.

DW Toolkit là gì?

Một bộ skills + workflow methodology chạy trên Claude Code CLI — biến AI thành một team member có quy trình làm việc bài bản.

⚙️

Config-driven

1 file YAML điều khiển toàn bộ behavior

🔌

Platform-agnostic

Methodology thuần, adapter cho từng AI platform

🎭

Multi-role

AI đóng vai BA, Dev, TL, QC, PM theo context

Config-driven là key insight: không hard-code workflow, project nào cũng config khác nhau. Platform-agnostic: logic nằm trong WORKFLOW.md, không phải trong Claude-specific syntax. Multi-role: cùng 1 agent có thể switch vai tùy phase của workflow.

Kiến Trúc

Cấu trúc thư mục

your-project/
├── CLAUDE.md          # Project instructions
├── .claude/
│   ├── skills/        # /dw-* commands
│   │   ├── dw-flow/
│   │   ├── dw-plan/
│   │   └── ...
│   ├── rules/         # Code style, workflow
│   └── templates/     # Task doc templates
└── .dw/
    ├── config/
    │   └── dw.config.yml  # ← Main config
    ├── tasks/         # Task documentation
    │   └── [task]/
    │       ├── context.md
    │       ├── plan.md
    │       └── progress.md
    └── core/          # Methodology docs

Config file

workflow:
  default_depth: standard

team:
  roles: [dev, techlead, qc]

tracking:
  estimation: true
  log_work: true

quality:
  test_command: "npm test"
  lint_command: "npm run lint"
  block_on_fail: true

💡 Đọc config đầu tiên trước mọi action

.claude/ là Claude Code-specific: skills, rules, templates. .dw/ là platform-agnostic: config, task docs, core methodology. dw.config.yml là source of truth cho mọi behavior.

Workflow Depths

AI tự assess scope, chọn depth phù hợp:

Depth Khi nào Phases
quick ≤2 files, hotfix, familiar module Understand → Execute → Close
standard 3-5 files, module mới Tất cả 6 phases
thorough 6+ files, API/DB/security changes Full + arch-review + test-plan

6 Core Phases

Initialize → Understand → Plan → Execute → Verify → Close
Default depth từ config, nhưng AI có thể override theo từng task. Quick: hotfix lúc 2am không cần full ceremony. Thorough: database migration, auth changes, public API changes. 6 phases = Init, Understand (research), Plan, Execute, Verify (review), Close (commit).

/dw-flow — Orchestrator

Một command, toàn bộ workflow từ A-Z với human checkpoints

Dùng như thế nào

# Standard workflow
/dw-flow add-user-authentication

# Override depth
/dw-flow DW_DEPTH=quick fix-login-bug

# Thorough với full ceremony
/dw-flow DW_DEPTH=thorough redesign-payment-api

Human Checkpoints (Gates)

  • GATE A: Confirm scope sau Research
  • GATE C: Approve plan ← HARD GATE
  • GATE D: Approve changes sau Review

Thorough Pipeline

Task Init
↓
Requirements
↓
GATE A
↓
Research + Arch Review
↓
Plan
↓
GATE C ← HARD
↓
Execute + Review
↓
Commit
Gate C là quan trọng nhất: AI không được execute nếu chưa có plan approval. User có thể gõ "revise: [feedback]" tại bất kỳ gate nào để AI chỉnh lại. Gõ "stop" để dừng workflow bất kỳ lúc nào.

Tips khi dùng DW Toolkit

🎯
Dùng /dw-flow cho tasks mới — Đừng nhảy thẳng vào execute
📝
Đọc plan trước khi approve — Gate C là checkpoint quan trọng nhất
🔄
Dùng "revise: [feedback]" — Nếu output chưa đúng, iterate tại gate
💾
Commit sau mỗi subtask — Small commits, dễ rollback
📊
Track effort thực tế — /dw-log-work để cải thiện estimate dần
🚀
Override depth khi cần — DW_DEPTH=quick cho hotfixes
Tip quan trọng nhất: đọc plan kỹ tại Gate C. Đây là lúc bạn control được AI direction. "Revise" là powerful: bạn có thể request changes mà không phải restart toàn bộ flow. Effort tracking giúp team estimate tốt hơn theo thời gian.

DW Toolkit — Điểm yếu cần lưu ý

Honest assessment: những gì chưa tốt và cần team chú ý

⏱️

Thorough depth = nhiều ceremony

12 phases cho thorough có thể quá nặng cho tasks nhỏ. Team cần discipline để chọn đúng depth, không áp dụng máy móc.

👉 Override: DW_DEPTH=quick cho hotfixes, config default_depth: standard cho projects nhỏ
🔌

Phụ thuộc vào Claude Code CLI

Hiện tại DW Toolkit optimize cho Claude Code. Nếu team switch AI tool (Cursor, Copilot…), cần adaptation work.

👉 Methodology trong WORKFLOW.md là platform-agnostic. Adapters có thể build cho platforms khác.
📚

Learning curve ban đầu

Team mới cần thời gian làm quen với skills, gates, config. Estimate đầu tiên thường không chính xác.

👉 Bắt đầu với 1-2 devs pilot, document learnings, rồi roll out dần cho cả team.
📝

Task docs có thể stale

Nếu không maintain đều, context/plan/progress files bị lạc hậu — mất đi giá trị handoff.

👉 Rule: luôn update progress trước khi end session. /dw-handoff giúp automate việc này.
🧪

Quality chỉ tốt khi config đúng

Nếu test_command và lint_command bỏ trống, quality gates không có tác dụng.

👉 Setup dw.config.yml với đúng commands ngay từ đầu. /dw-config-validate để kiểm tra.
Transparency quan trọng: không có tool nào perfect. Biết điểm yếu để dùng đúng chỗ. Ceremony concern: một dev từng phàn nàn "dw-flow mất 30 phút cho một bug fix 5 dòng" → lesson learned, dùng DW_DEPTH=quick cho cases đó. Stale docs: giống như stale tests, stale docs worse than no docs. Cần team culture để maintain. Platform dependency: Anthropic đang develop Agent SDK để cho phép build workflows phức tạp hơn nữa.
